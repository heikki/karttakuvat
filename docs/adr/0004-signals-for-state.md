# State held in `@lit-labs/signals`

Application state lives in `@lit-labs/signals` stores under `src/client/common/` (`data`, `edits`, `selection`, `view-state`, `interaction-mode`). Lit components opt into reactivity via the `SignalWatcher` mixin and read signals in `render()`; non-component modules use `effect()` from `@common/signals` for module-level reactions. Considered: per-element `@state` only (forces prop-drilling for cross-feature state), `@lit/context` only (good for handing handles down a tree but awkward for verb-style commands and computed derivations), and a hand-rolled pub/sub. Signals won because they're a single primitive that handles both component reactivity and module-level effects, and because they compose cleanly with `urlSignal()` for URL-bound state.
